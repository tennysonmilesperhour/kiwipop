# Deploying Kiwi Pop

End-to-end deployment of the Kiwi Pop Next.js storefront to **Vercel** with
**Supabase** for database/auth and **Stripe** for payments. Follow these
steps in order — most failures are environment-variable related.

## 0. Prerequisites

You need the following accounts and resources already created:

- A Supabase project (Project URL + anon key + service-role key on hand)
- A Stripe account (live keys ready — only switch to live keys when you're
  ready to take real money)
- A Vercel account, with this repository connected as a project
- (Optional) A custom domain ready to point at Vercel

## 1. Apply Supabase migrations

The schema lives in `supabase/migrations/`. Run them against your Supabase
project before the app first hits the DB.

**Easiest path (Dashboard):**

1. Open your Supabase project → SQL Editor.
2. Open `supabase/migrations/001_initial_schema.sql`, paste, run.
3. Open `supabase/migrations/002_rls_and_triggers.sql`, paste, run.

**With the Supabase CLI:**

```bash
supabase link --project-ref <your-project-ref>
supabase db push
```

After running `002_rls_and_triggers.sql` you should be able to verify with:

```sql
select tablename, rowsecurity from pg_tables where schemaname='public' order by tablename;
```

Every table should have `rowsecurity = true`.

## 2. Promote yourself to admin

Until you have an admin user, the `/admin/*` routes will redirect everyone
to `/`.

1. Sign up at `https://<your-domain>/auth/signup` (or run locally first).
2. In the Supabase SQL editor:
   ```sql
   update public.profiles set role = 'admin' where email = 'you@example.com';
   ```

The `profiles` row is auto-created by the `on_auth_user_created` trigger
shipped in migration 002.

## 3. Configure Vercel environment variables

In Vercel → Project Settings → Environment Variables, add the following
for **Production** (and Preview if you want preview deploys to be functional):

| Name | Value | Scope |
| ---- | ----- | ----- |
| `NEXT_PUBLIC_SUPABASE_URL` | `https://<ref>.supabase.co` | Public |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | anon key from Supabase | Public |
| `SUPABASE_SERVICE_ROLE_KEY` | service-role key from Supabase | **Server-only** |
| `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY` | `pk_live_…` | Public |
| `STRIPE_SECRET_KEY` | `sk_live_…` | **Server-only** |
| `STRIPE_WEBHOOK_SECRET` | `whsec_…` (from step 4) | **Server-only** |
| `NEXT_PUBLIC_SITE_URL` | `https://<your-domain>` | Public |

Values marked "Server-only" must NOT have `NEXT_PUBLIC_` in the name. Vercel
will only inject them into server-side bundles.

## 4. Configure the Stripe webhook

Once Vercel has assigned you a deploy URL (or your custom domain is wired):

1. Stripe Dashboard → Developers → Webhooks → **Add endpoint**.
2. Endpoint URL: `https://<your-domain>/api/webhooks/stripe`
3. Events to listen for:
   - `checkout.session.completed`
   - `payment_intent.payment_failed`
   - `charge.refunded`
4. Click **Add endpoint**, then copy the **Signing secret** (starts with
   `whsec_`). Paste it into Vercel as `STRIPE_WEBHOOK_SECRET` and redeploy.

## 5. Custom domain

Vercel → Domains → add the domain, follow the DNS instructions. Once it
resolves, update `NEXT_PUBLIC_SITE_URL` to the canonical https URL and
redeploy. Stripe success/cancel URLs use this, as does the sitemap.

## 6. Smoke test

After deploy:

- **Public**: `/`, `/products/<id>`, `/cart` all render. Adding to cart and
  refreshing the page should preserve the cart.
- **Auth**: sign up → confirm a `profiles` row exists in Supabase
  (`select * from profiles where email='…'`). It should have `role='customer'`.
- **Checkout**: with a Stripe test card (`4242 4242 4242 4242`, any future
  expiry, any CVC), place an order. After Stripe redirects to
  `/checkout/success`, the corresponding `orders` row in Supabase should
  flip to `status='paid'` within a few seconds (webhook).
- **Admin**: promote yourself, hit `/admin/dashboard`, then on each admin
  page perform one mutation (add a product, adjust inventory, change an
  order status, record an expense, …). Refresh; changes should persist.
- **Refund**: change a paid order's status to `cancelled`. Stripe should
  show the refund and a row should appear in `returns`.

## 7. Headers / SEO check

After deploy, verify from your terminal:

```bash
curl -sI https://<your-domain>/ | grep -iE "strict-transport-security|x-content-type|x-frame|referrer-policy"
curl -s https://<your-domain>/robots.txt
curl -s https://<your-domain>/sitemap.xml | head -40
```

Both robots.txt and sitemap.xml are dynamically generated by Next.js.

## Guest checkout

The storefront supports both guest and authenticated checkout out of the box:

- **Guest**: shopper provides email + shipping address, no account created.
  Order is stored with `user_id = NULL` and only retrievable via the
  order-id link emailed to them.
- **Logged in**: if the shopper has signed in before checkout, the order's
  `user_id` is auto-populated server-side from their session.

Either way the order-confirmation page (`/order-confirmation/<id>`) is
server-rendered with the service-role key, so guests can view their
receipt by URL without auth.

## Known gaps (not launch blockers, but track these)

- Bill-of-materials editor: tables exist, RLS in place, no UI yet
  (the schema supports it; you'd add a small editor inside Manufacturing
  if you start tracking ingredient consumption per batch).
- Wholesale **quote PDF generation**: quotes table exists, no UI yet
  (the wholesale page covers tier pricing setup and account approvals).
- Email confirmations on order paid (wire Resend or Supabase Auth SMTP).
- Real product imagery / OG image (`public/og-image.png` referenced in
  metadata but not committed — drop in your hero image).
- Lighthouse pass (target ≥ 85 mobile performance, 95 accessibility/SEO).
